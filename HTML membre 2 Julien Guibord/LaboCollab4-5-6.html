Introduction

Objectifs d'apprentissage

À la fin de ce laboratoire, vous serez capables de :

  • Connecter votre dépôt local à un dépôt distant hébergé sur GitHub
  • Ajouter des collaborateurs à un projet pour travailler ensemble
  • Effectuer des modifications de fichiers et enregistrer ces changements avec des commits
  • Comprendre ce qu'est un conflit de fusion et apprendre à le résoudre
  • Documenter votre travail à l'aide de captures d'écran pour constituer un rapport

Vocabulaire de base

  • GitHub : Un site web qui héberge des dépôts Git en ligne, facilitant le travail d'équipe
Conseil important : Prenez une capture d'écran à chaque étape importante mentionnée dans ce guide. Ces captures serviront de preuves de votre travail.

Partie 1 : Configuration initiale du projet

Dans cette première partie, un membre de l'équipe va créer le projet de base sur son ordinateur et le publier sur GitHub. Les autres membres pourront ensuite y accéder.

Étape 1.1 : Création du dépôt local

Personne responsable : Membre 1 (le chef d'équipe)

Le dépôt local est une copie de votre projet stockée directement sur votre ordinateur. Vous allez maintenant le créer.

Instructions détaillées :

  1. Sur votre ordinateur, créez un nouveau dossier (par exemple sur le Bureau ou dans Documents) et nommez-le projet-equipe-[votre-nom]
  2. Ouvrez le terminal (ou Git Bash si vous êtes sous Windows) et placez-vous dans ce dossier
  3. Tapez la commande suivante pour dire à Git de commencer à surveiller ce dossier :
git init
La commande git init transforme un dossier ordinaire en dépôt Git. C'est comme si vous disiez à Git : « Commence à surveiller ce dossier et garde une trace de tous les changements. »
  1. Ouvrez Visual Studio Code dans ce dossier (vous pouvez le faire en tapant code . dans le terminal)
  2. Dans VS Code, créez un nouveau fichier et nommez-le README.md
  3. Écrivez le texte suivant dans ce fichier :
# Projet de collaboration - Équipe [Nom de votre équipe] ## Description Ce projet nous permet d'apprendre à utiliser Git en équipe.
Le fichier README.md est un document qui explique ce qu'est votre projet. Le .md signifie « Markdown », un format de texte simple qui permet de faire des titres, des listes, etc.
  1. Sauvegardez le fichier.
  2. Retournez dans le terminal et tapez :
git add README.md
git commit -m "Initialisation du projet"

Étape 1.2 : Création des comptes GitHub et du dépôt distant

Personnes responsables : TOUS les membres de l'équipe

GitHub est une plateforme en ligne qui permet de stocker et de partager des projets Git. Chaque membre de l'équipe doit d'abord créer un compte personnel.

Important : Tous les membres doivent créer leur propre compte GitHub avant de continuer. Sans compte, vous ne pourrez pas collaborer sur le projet.

Instructions pour CHAQUE membre de l'équipe :

  1. Ouvrez votre navigateur web et allez sur : https://github.com
  2. Cliquez sur le bouton « Sign up » (S'inscrire) en haut à droite de la page
  3. Remplissez le formulaire d'inscription en fournissant :
    • Votre adresse courriel (utilisez votre courriel scolaire si possible)
    • Un mot de passe sécurisé (avec des lettres, des chiffres et des symboles)
    • Un nom d'utilisateur professionnel (évitez les surnoms trop informels car ce compte pourra vous servir plus tard dans vos études)
  4. Consultez votre boîte de réception et cliquez sur le lien de vérification envoyé par GitHub
  5. Complétez votre profil si GitHub vous le demande
  6. Notez bien votre nom d'utilisateur GitHub sur un papier ou dans un document, vous en aurez besoin

Instructions pour le Membre 1 uniquement (création du dépôt distant) :

Une fois que tous les membres ont créé leur compte, le Membre 1 peut créer le dépôt en ligne :

  1. Connectez-vous à votre compte GitHub
  2. Cliquez sur le bouton vert « New repository » (Nouveau dépôt) ou sur le symbole + en haut à droite, puis sélectionnez « New repository »

Étape 1.3 : Connexion du dépôt local au dépôt distant

Personne responsable : Membre 1

Maintenant que le dépôt local et le dépôt GitHub existent, il faut les relier ensemble pour pouvoir synchroniser les fichiers entre les deux.

Instructions :

  1. Sur la page de votre dépôt GitHub, copiez l'URL qui apparaît (elle se termine par .git)
  2. Retournez dans votre terminal et tapez la commande suivante en remplaçant [URL-de-votre-depot] par l'URL que vous venez de copier :
git remote add origin [URL-de-votre-depot]
Cette commande crée un lien entre votre dépôt local et le dépôt GitHub. Le mot origin est simplement un nom qui désigne ce dépôt distant.
  1. Envoyez maintenant votre premier commit vers GitHub avec cette commande :
git push -u origin master
Vocabulaire technique :
  • Dépôt distant : Une copie de votre projet stockée sur un serveur en ligne (ici, sur GitHub) et accessible à toute l'équipe
  • origin : Le nom par défaut que Git donne au dépôt distant principal
  • push : Envoyer vos modifications locales vers le dépôt distant (imaginez que vous « poussez » vos fichiers vers le cloud)

Partie 2 : Ajout des collaborateurs

Maintenant que le projet est en ligne sur GitHub, il faut inviter les autres membres de l'équipe pour qu'ils puissent y contribuer.

Étape 2.1 : Invitation des membres de l'équipe

Personne responsable : Membre 1

Instructions :

  1. Sur la page GitHub de votre dépôt, cliquez sur l'onglet « Settings » (Paramètres) en haut de la page
  2. Dans le menu de gauche, trouvez et cliquez sur « Collaborators » (Collaborateurs)
  3. GitHub vous demandera probablement de confirmer votre identité en entrant votre mot de passe
  4. Cliquez sur le bouton vert « Add people » (Ajouter des personnes)
  5. Dans la boîte de recherche qui apparaît, tapez le nom d'utilisateur GitHub (idéalement son mail) d'un de vos coéquipiers
  6. Sélectionnez le bon compte dans les résultats et cliquez sur « Add [nom] to this repository »
  7. Répétez les étapes 4 à 6 pour chaque membre de l'équipe
Vérifiez bien que vos coéquipiers vous ont donné leur nom d'utilisateur GitHub exact. Attention aux fautes de frappe, aux majuscules et aux caractères spéciaux !
Capture d'écran requise : Prenez une capture d'écran de la page « Collaborators » montrant les invitations envoyées (avec les noms d'utilisateur de vos coéquipiers visibles).

Étape 2.2 : Acceptation de l'invitation et clonage du projet

Personnes responsables : Membres 2 et 3

Les membres invités doivent maintenant accepter l'invitation et obtenir une copie du projet sur leur propre ordinateur.

Instructions :

  1. Consultez votre boîte de réception email (l'invitation de GitHub y a été envoyée)
  2. Cliquez sur le lien dans le courriel OU connectez-vous à GitHub et cliquez sur l'icône de notification (en forme de cloche) en haut à droite
  3. Cliquez sur le bouton « Accept invitation » (Accepter l'invitation)
  4. Vous serez redirigés vers la page du dépôt du projet
  5. Sur cette page, cliquez sur le bouton vert « Code »
  6. Assurez-vous que l'onglet « HTTPS » est sélectionné, puis copiez l'URL qui s'affiche
  7. Sur votre ordinateur, créez un dossier de travail nommé Laboratoire-Git (vous pouvez le créer sur le Bureau ou dans Documents)
  8. Ouvrez le terminal (ou Git Bash) et déplacez-vous dans ce nouveau dossier :
cd Chemin/vers/Laboratoire-Git
Astuce Windows : Vous pouvez faire un clic droit directement dans le dossier « Laboratoire-Git » et sélectionner « Git Bash Here » pour ouvrir le terminal au bon endroit.
  1. Tapez maintenant la commande suivante pour créer une copie complète du projet sur votre ordinateur (remplacez [URL-du-depot] par l'URL copiée plus tôt) :
git clone [URL-du-depot]
  1. Une fois le clonage terminé, entrez dans le dossier du projet :
cd projet-equipe-collaboration
Qu'est-ce que le clonage ?
La commande git clone crée une copie complète d'un dépôt distant sur votre ordinateur. Cette copie inclut tous les fichiers du projet ainsi que tout l'historique des modifications. C'est comme télécharger un dossier complet depuis internet.

Partie 3 : Première collaboration sans conflit

Dans cette partie, chaque membre va créer son propre fichier. Comme les fichiers seront différents, il n'y aura pas de conflit. Cela vous permettra de comprendre le processus de base pour partager votre travail.

Étape 3.1 : Création de fichiers par différents membres

Objectif : Chaque membre va créer son propre fichier pour s'exercer aux commandes Git de base sans risque de conflit.

Instructions pour le Membre 1 :

  1. Ouvrez Visual Studio Code dans le dossier de votre projet
  2. Créez un nouveau fichier et nommez-le membre1.txt
  3. Écrivez le contenu suivant dans ce fichier :
Nom : [Votre nom complet] Rôle : Chef d'équipe Responsabilités : Coordination du projet et gestion du dépôt GitHub
  1. Sauvegardez le fichier (Ctrl+S ou Cmd+S)
  2. Dans le terminal, enregistrez et envoyez ce fichier vers GitHub :
git add membre1.txt git commit -m "Ajout des informations du membre 1" git push

Instructions pour le Membre 2 :

  1. Avant de commencer à travailler, récupérez toujours les dernières modifications de l'équipe :
git pull
La commande git pull télécharge les modifications que vos coéquipiers ont envoyées sur GitHub. C'est l'inverse de git push. Prenez l'habitude de faire un git pull avant de commencer à travailler !
  1. Ouvrez Visual Studio Code dans le dossier du projet
  2. Créez un nouveau fichier nommé membre2.txt
  3. Écrivez le contenu suivant :
Nom : [Votre nom complet] Rôle : Développeur Responsabilités : Développement et tests du code
  1. Sauvegardez le fichier
  2. Enregistrez et envoyez vos modifications :
git add membre2.txt git commit -m "Ajout des informations du membre 2" git push

Instructions pour le Membre 3 :

  1. Récupérez d'abord les modifications récentes :
git pull
  1. Ouvrez Visual Studio Code
  2. Créez un fichier nommé membre3.txt
  3. Écrivez le contenu suivant :
Nom : [Votre nom complet] Rôle : Testeur Responsabilités : Tests et validation du travail d'équipe
  1. Sauvegardez le fichier
  2. Enregistrez et envoyez :
git add membre3.txt git commit -m "Ajout des informations du membre 3" git push
Bonne pratique : Toujours faire un git pull avant de commencer à travailler. Cela vous permet de récupérer les modifications de vos coéquipiers et d'éviter les conflits.

Partie 4 : Situation de conflit simple

Un conflit se produit lorsque deux personnes modifient la même partie d'un même fichier en même temps. C'est une situation normale dans le travail d'équipe, et Git vous aide à la gérer.

Étape 4.1 : Création volontaire d'un conflit

Objectif : Comprendre ce qui se passe quand deux personnes modifient le même fichier simultanément. Les conflits sont normaux et il est important de savoir les résoudre.
Scénario : Les membres 1 et 2 vont tous les deux modifier le fichier README.md en même temps, sans se coordonner.

Instructions pour le Membre 1 :

  1. Ouvrez le fichier README.md dans Visual Studio Code
  2. Ajoutez le texte suivant à la fin du fichier :
## Membres de l'équipe - Chef d'équipe : [Votre nom]
  1. Sauvegardez le fichier, puis enregistrez et envoyez les modifications :
git add README.md git commit -m "Ajout de la section chef d'équipe" git push

Instructions pour le Membre 2 :

Important : NE FAITES PAS de git pull avant de commencer ! C'est volontaire pour créer un conflit et apprendre à le gérer.
  1. Ouvrez le fichier README.md dans Visual Studio Code
  2. Ajoutez le texte suivant à la fin du fichier :
## Membres de l'équipe - Développeur : [Votre nom]
  1. Sauvegardez et enregistrez vos modifications :
git add README.md git commit -m "Ajout de la section développeur"
  1. Essayez maintenant d'envoyer vos modifications vers GitHub :
git push
Résultat attendu : Git va rejeter votre tentative d'envoi avec un message d'erreur. Ce message vous informe que le dépôt distant contient des modifications que vous n'avez pas encore récupérées. Vous verrez quelque chose comme :

! [rejected] main -> main (fetch first)
Pourquoi cette erreur ?
Git vous empêche d'écraser le travail du Membre 1. Il vous demande d'abord de récupérer ses modifications avec git pull, puis de fusionner les deux versions.

Insérez ici votre capture d'écran

Étape 4.2 : Résolution du conflit

Personne responsable : Membre 2

Instructions détaillées :

  1. Récupérez les modifications du dépôt distant :
git pull
Résultat : Git va détecter un conflit car les deux versions du fichier ont modifié la même zone. Un message vous l'indiquera.
  1. Ouvrez le fichier README.md dans Visual Studio Code. Vous verrez quelque chose comme ceci :
## Membres de l'équipe <<<<<<< HEAD - Développeur : [Nom Membre 2] ======= - Chef d'équipe : [Nom Membre 1] >>>>>>> [identifiant du commit]

Explication des symboles de conflit :

  • <<<<<<< HEAD : Marque le début de vos modifications locales (ce que vous avez écrit)
  • ======= : Sépare vos modifications de celles provenant de GitHub
  • >>>>>>> [id] : Marque la fin des modifications provenant du dépôt distant (ce qu'a écrit le Membre 1)
  1. Pour résoudre le conflit, modifiez le fichier pour combiner les deux versions. Remplacez tout le bloc de conflit (y compris les marqueurs) par :
## Membres de l'équipe - Chef d'équipe : [Nom Membre 1] - Développeur : [Nom Membre 2]
  1. Vérifiez bien que vous avez supprimé tous les marqueurs de conflit (<<<<<<<, =======, >>>>>>>)
  2. Sauvegardez le fichier et finalisez la résolution :
git add README.md git commit -m "Résolution du conflit sur README.md" git push
Conseil : Lors de la résolution d'un conflit, assurez-vous de garder les informations importantes des deux versions. Communiquez avec votre équipe si vous n'êtes pas certain de ce qu'il faut conserver.

Partie 5 : Collaboration sur un fichier HTML

Dans cette partie, vous allez créer une page web simple en HTML et apprendre à collaborer sur ce type de fichier.

Étape 5.1 : Création du fichier HTML de base

Personne responsable : Membre 1

Instructions :

  1. Ouvrez Visual Studio Code
  2. Créez un nouveau fichier nommé index.html
  3. Copiez le code HTML suivant dans ce fichier :
<!DOCTYPE html> <html lang="fr"> <head> <meta charset="UTF-8"> <meta name="viewport" content="width=device-width, initial-scale=1.0"> <title>Projet d'équipe</title> </head> <body> <h1>Notre projet collaboratif</h1> <!-- Section à compléter par les membres --> </body> </html>
Qu'est-ce que le HTML ?
HTML (HyperText Markup Language) est le langage utilisé pour créer des pages web. Les balises comme <h1> (titre) ou <body> (corps de la page) donnent une structure au contenu.
  1. Sauvegardez le fichier
  2. Enregistrez et envoyez ce fichier vers GitHub :
git add index.html git commit -m "Création du fichier HTML de base" git push

Partie 6 : Utilisation des branches

Les branches permettent de travailler sur des fonctionnalités séparées sans affecter le projet principal.

Étape 6.1 : Création d'une branche pour une nouvelle fonctionnalité

Objectif : Apprendre à créer et travailler sur une branche séparée pour ajouter du style CSS à la page HTML.

Instructions pour le Membre 1 :

  1. Créez une nouvelle branche pour ajouter un style CSS :
git checkout -b ajout-style
La commande git checkout -b crée une nouvelle branche ET vous place dessus en une seule étape. Vous travaillez maintenant sur la branche "ajout-style" au lieu de "main".
  1. Ouvrez VS Code et créez un nouveau fichier nommé style.css
  2. Copiez le code CSS suivant dans ce fichier :
body { font-family: Arial, sans-serif; margin: 20px; background-color: #f0f0f0; } h1 { color: #333; text-align: center; } section { background-color: white; padding: 20px; margin: 20px 0; border-radius: 8px; box-shadow: 0 2px 5px rgba(0,0,0,0.1); }
  1. Sauvegardez le fichier
  2. Modifiez maintenant le fichier index.html pour lier cette feuille de style. Ajoutez cette ligne dans la section <head> :
<link rel="stylesheet" href="style.css">
  1. Sauvegardez, enregistrez sur votre branche et envoyez-la vers GitHub :
git add . git commit -m "Ajout du fichier CSS avec styles de base" git push -u origin ajout-style
La commande git add . ajoute tous les fichiers modifiés d'un coup. Le paramètre -u origin ajout-style crée la branche sur GitHub et établit un lien entre votre branche locale et la branche distante.
Capture d'écran requise : Terminal montrant la création de la branche et le push réussi.
Insérez ici votre capture d'écran

Étape 6.2 : Fusion de la branche (merge)

Personne responsable : Membre 1

Maintenant que le travail sur la branche est terminé, il faut la fusionner avec la branche principale pour que tout le monde bénéficie des nouveaux styles.

Instructions :

  1. Revenez sur la branche principale :
git checkout main
  1. Fusionnez votre branche "ajout-style" avec la branche principale :
git merge ajout-style
La commande git merge intègre tous les changements d'une branche dans une autre. Ici, vous intégrez les fichiers CSS créés sur la branche "ajout-style" dans la branche "main".
  1. Envoyez les modifications fusionnées vers GitHub :
git push
  1. Tous les membres peuvent maintenant récupérer le nouveau style :
git pull
Après avoir récupéré les modifications, ouvrez le fichier index.html dans votre navigateur pour voir la page web avec les nouveaux styles appliqués !

Partie 7 : Situations réalistes de collaboration

Cette partie vous présente des situations courantes que vous rencontrerez en travaillant en équipe avec Git.

Situation 1 : Modification simultanée de fichiers différents

Objectif : Démontrer que Git gère très bien les modifications effectuées sur des fichiers différents, même si elles sont faites en même temps.

Instructions :

  • Membre 1 : Ouvrez membre1.txt dans VS Code et ajoutez une ligne décrivant une compétence que vous possédez
  • Membre 2 : Ouvrez membre2.txt dans VS Code et ajoutez une ligne décrivant une compétence que vous possédez
  • Les deux membres : Enregistrez et envoyez vos modifications en même temps (sans faire de git pull au préalable)
Résultat attendu : Aucun conflit ne se produira car vous avez modifié des fichiers différents. Le premier qui envoie ses modifications réussira immédiatement. Le second devra faire un git pull, puis pourra envoyer sans problème car Git combinera automatiquement les changements.
Capture d'écran requise (les deux membres) : Terminal montrant le pull et le push réussis.

Situation 2 : Oubli de faire un pull

Objectif : Comprendre pourquoi il est important de toujours faire git pull avant de commencer à travailler.

Instructions :

  1. Membre 1 : Modifiez le fichier README.md en ajoutant une nouvelle ligne dans la description
  2. Membre 1 : Enregistrez et envoyez vos modifications
  3. Membre 2 : Sans faire de git pull, modifiez aussi README.md en ajoutant une ligne différente
  4. Membre 2 : Enregistrez et essayez d'envoyer
  5. Membre 2 : Faites un git pull, résolvez le conflit si nécessaire, puis envoyez
Leçon importante : Prenez l'habitude de toujours commencer votre journée de travail par un git pull pour récupérer les dernières modifications de l'équipe. Cela évite beaucoup de conflits inutiles.

Situation 3 : Annulation d'un commit

Objectif : Apprendre à revenir en arrière en cas d'erreur dans un commit.

Instructions :

  1. Un membre au choix : Créez un fichier nommé erreur.txt avec du contenu quelconque
  2. Enregistrez ce fichier avec un commit :
git add erreur.txt git commit -m "Ajout d'un fichier par erreur"
  1. Pour annuler le dernier commit sans perdre les modifications :
git reset --soft HEAD~1
Avec l'option --soft, le commit est annulé mais le fichier reste dans votre dossier et dans la zone de préparation (prêt à être enregistré à nouveau si vous le souhaitez).
  1. Pour annuler complètement le dernier commit et supprimer les modifications :
git reset --hard HEAD~1
Attention : L'option --hard supprime définitivement vos modifications ! Utilisez cette commande avec beaucoup de précaution et seulement si vous êtes certain de vouloir tout effacer.
Explication de HEAD~1 :
HEAD représente votre position actuelle dans l'historique. HEAD~1 signifie "un commit avant HEAD", donc le commit précédent.
Capture d'écran requise : Terminal montrant l'annulation du commit. Utilisez git log avant et après l'annulation pour voir la différence.

Situation 4 : Visualisation de l'historique

Objectif : Apprendre à consulter l'historique complet des modifications du projet.

Instructions (tous les membres) :

  1. Pour voir l'historique complet des commits :
git log
Cette commande affiche tous les commits, du plus récent au plus ancien, avec l'auteur, la date et le message de chaque commit.
  1. Pour une version condensée et plus facile à lire :
git log --oneline
  1. Pour voir l'historique avec un graphique montrant les branches :
git log --oneline --graph --all
Le format --oneline --graph --all est très utile pour visualiser comment les branches se divisent et se fusionnent au fil du temps. C'est comme une carte de votre projet.
